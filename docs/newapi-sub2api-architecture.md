# new-api + Sub2API 混合架构说明

适用目标：

- 对外只暴露一个统一 API 入口
- 同时接入国内外大模型
- 保留 Sub2API 的国外账号池能力
- 由 new-api 统一负责用户、计费、限流和管理后台
- 尽量降低双网关串联带来的复杂度

本文档给出推荐职责划分、模型路由和上线注意事项。

## 1. 总体结论

推荐架构：

`客户端 -> new-api -> 国内上游`

`客户端 -> new-api -> Sub2API -> 国外上游`

建议角色分工：

- `new-api` 作为对外总网关
- `Sub2API` 作为内层国外账号池网关
- 国内模型直接由 `new-api` 对接，不经过 `Sub2API`

不推荐把所有流量都先打到 `Sub2API`，再回到 `new-api`。

## 2. 角色定位

### 2.1 new-api 负责什么

`new-api` 负责所有对外能力：

- 统一域名和 API 入口
- 用户体系、用户分组、权限控制
- 对外 API Key 发放
- 外层计费、余额和配额扣减
- 外层限流和风控
- 模型路由
- 日志、审计和管理后台
- 国内模型直连
- 将需要账号池的国外模型转发给 `Sub2API`

一句话：`new-api` 是产品入口和主控制面。

### 2.2 Sub2API 负责什么

`Sub2API` 负责国外账号池相关能力：

- 上游 OAuth / API Key 账号维护
- 账号池调度
- sticky session
- 并发控制
- 国外上游请求转发
- Claude / OpenAI / Gemini 等国外模型订阅池能力

一句话：`Sub2API` 是内部国外账号池网关。

## 3. 为什么这样拆

这样拆有四个直接好处：

1. 对外只有一个入口，客户端和用户侧配置最简单
2. 国内模型不多绕一层，延迟和故障面都更小
3. 国外号池继续由 `Sub2API` 处理，不需要在 `new-api` 内重复造轮子
4. 后续替换 `Sub2API` 或调整国外上游，不需要改客户端

## 4. 推荐流量路径

### 4.1 国内模型

推荐流量：

`客户端 -> new-api -> 国内厂商官方接口或国内代理`

适合直接走 `new-api` 的模型类型：

- 阿里通义 / Qwen
- 智谱 / GLM
- 百度文心千帆
- 腾讯混元
- 火山方舟 / 豆包
- Moonshot
- DeepSeek
- 其他国内兼容 OpenAI / Claude / Gemini 协议的服务

### 4.2 国外模型

推荐流量：

`客户端 -> new-api -> Sub2API -> 国外上游`

适合走 `Sub2API` 的模型类型：

- Claude / Claude Code
- OpenAI / Codex 订阅型能力
- Gemini 订阅池
- 其他明确依赖账号池、会话粘性或订阅凭据的国外模型

## 5. 职责边界

### 5.1 应该放在 new-api 的能力

建议只在 `new-api` 做这些事情：

- 对外用户认证
- 对外 API Key 管理
- 主计费系统
- 主余额系统
- 主限流策略
- 模型命名、展示和统一路由
- 国内外模型统一监控入口
- 管理后台和审计日志

### 5.2 应该留在 Sub2API 的能力

建议只在 `Sub2API` 保留这些事情：

- 国外账号池维护
- 上游认证
- 上游 session stickiness
- 账号级并发保护
- 上游特定头部和行为兼容

### 5.3 不建议两边同时做的事情

尽量不要让 `new-api` 和 `Sub2API` 同时完整处理以下能力：

- 主计费
- 主余额
- 主限流
- 主重试
- 主熔断

否则会变成双控制面，排障和计费对账都会变难。

## 6. Sub2API 建议运行方式

推荐把 `Sub2API` 收缩成“内部专用网关”，不要让它成为对外产品入口。

建议做法：

- 使用 `RUN_MODE=simple`
- 生产环境再加 `SIMPLE_MODE_CONFIRM=true`
- 不直接给真实外部用户开放 `Sub2API`
- 只允许 `new-api` 访问它
- 仅保留内部使用的少量 API Key
- 将并发和速率控制调宽，避免和 `new-api` 抢主控制权

注意：

- `Sub2API` 可以弱化 SaaS 能力和 billing 流程
- 但它不能完全退化成纯机械转发器
- 因为账号池调度、sticky session、上游认证本身就是它的核心价值

## 7. 典型部署拓扑

推荐最小部署拓扑：

`客户端 -> Nginx -> new-api`

`new-api -> 国内上游`

`new-api -> 内网 Sub2API -> 国外上游`

推荐原则：

- `new-api` 对公网暴露
- `Sub2API` 尽量只放内网
- `new-api` 与 `Sub2API` 尽量同机或同地域部署
- 尽量避免跨地域多一跳

## 8. 可能的性能影响

串联架构一定会多一跳：

`客户端 -> new-api -> Sub2API -> 上游`

主要代价：

- 多一次网络往返
- 多一层应用处理
- 流式响应链路更长

一般情况下，真正更值得关注的不是那一点延迟，而是：

- 流式首包延迟
- 中途断流
- 错误码被二次包装
- header 丢失或被改写
- 双层重试导致的问题放大

因此，部署重点不是“追求零损耗”，而是“控制复杂度”。

## 9. 接入时的技术注意点

### 9.1 Header 透传

重点检查这些内容是否会在两层之间丢失或被改写：

- `Authorization`
- 上游厂商特定 header
- 流式响应相关 header
- 会话粘性相关 header
- Claude / Codex / Gemini 专用 header

### 9.2 超时配置

两层网关串联时，需要统一：

- 读取超时
- 写入超时
- 流式连接超时
- 反向代理超时

避免外层比内层更短，导致请求被过早切断。

### 9.3 重试策略

建议只让一层承担主要重试职责。

否则容易出现：

- 同一请求重复执行
- 额度重复扣减
- 上游压力被放大

### 9.4 计费与配额

建议以 `new-api` 为唯一对外计费入口。

`Sub2API` 内部只保留最小必要的保护逻辑，不作为对外主账本。

## 10. 推荐模型路由策略

下面给出一版可直接执行的默认策略。

### 10.1 走 new-api 直连

- `qwen-*`
- `glm-*`
- `ernie-*`
- `hunyuan-*`
- `doubao-*`
- `moonshot-*`
- `deepseek-*`
- 其他国内源

### 10.2 走 Sub2API

- `claude-*`
- `gpt-*` 中依赖订阅池或账号池的模型
- `codex-*`
- `gemini-*` 中依赖账号池的模型

如果某个国外模型本身就有稳定官方 API Key，且你不需要账号池，也可以考虑直接走 `new-api`，不要强制套 `Sub2API`。

## 11. 不推荐的架构

### 11.1 所有模型都先走 Sub2API

不推荐：

`客户端 -> new-api -> Sub2API -> 国内外所有上游`

问题：

- 国内模型无谓多一跳
- 依赖点更多
- 排障更复杂

### 11.2 双方同时做完整产品能力

不推荐：

- `new-api` 做一套用户/计费/限流
- `Sub2API` 再做一套用户/计费/限流

问题：

- 权限和额度边界不清楚
- 数据和账本容易打架
- 运维复杂度显著上升

## 12. 上线前检查清单

上线前至少确认以下内容：

- `new-api` 是唯一公网 API 入口
- `Sub2API` 不直接暴露给外部用户
- 国内模型没有被错误转发到 `Sub2API`
- 国外模型已按账号池需求正确路由
- 两层超时一致
- 流式响应没有被缓冲
- 只有一层负责主要计费
- 只有一层负责主要限流
- 关键 header 已验证透传
- 401 / 429 / 5xx 返回行为已实际联调

## 13. 最终建议

如果目标是：

- 同时支持国内外模型
- 对外统一一个 API
- 国外模型继续使用账号池

推荐最终方案就是：

- `new-api` 做总网关
- `Sub2API` 做国外账号池内层
- 国内模型直连，不经过 `Sub2API`

这套方案的核心不是“把所有事情都塞进一个系统”，而是“让每一层只做自己最擅长的事”。
